## Signal Aborted

**Root asymmetry: the web API client never catches `AbortError`; the mobile client does.**

- [nextstep-mobile/src/api/client.ts:71-89](nextstep-mobile/src/api/client.ts:71) wraps `fetch` in a `try/catch`, detects `err.name === 'AbortError'`, and rethrows a friendly `ApiRequestError` (`'Request timed out...'` / `'The school portal did not respond in time...'`). This was already correct.
- [lib/api.ts:51-69](lib/api.ts:51) (web, **before fix**) wrapped `fetch` in `try { … } finally { clearTimeout(timeout) }` — **no `catch` block at all**. Any abort (timeout or otherwise) threw a raw, uncaught `DOMException: "The operation was aborted"` / `"signal is aborted without reason"` straight out of `request()` to whichever caller invoked it. Several callers (e.g. `app/(app)/colleges/page.tsx:85`) do a bare `.catch(() => {})`, so the failure was silently swallowed as "nothing loaded"; others surfaced the raw abort message directly — this is the literal "signal aborted" text users were seeing.
- Web also used a single flat 30s timeout ([lib/api.ts:54](lib/api.ts:54), before fix) for **every** endpoint, including the ones that proxy live HAC/PowerSchool scraping (`portalLoginHAC`, `portalSyncProfile`, `portalGrades`, `portalTranscript`, etc.). Mobile already special-cases these with a 45s tier (`isScrapingEndpoint`, see `nextstep-mobile/src/api/gradesApi.ts`) specifically because "a short uniform timeout caused scraping failures" (per the file's own header comment). Web had no equivalent tiering.
- No `Promise.all` fail-fast bug was found — every parallel-load site (`app/(app)/settings/page.tsx:309`, `app/(app)/dashboard/page.tsx:170`, `nextstep-mobile/src/screens/CollegesScreen.tsx:167`, etc.) either accepts the whole-batch semantics intentionally or already guards each sub-promise with its own `.catch()`.
- No unconditional useEffect-abort-on-unmount pattern was found; the abort controllers in both clients are purely per-request timeouts, not tied to component lifecycle/navigation.

**Fix applied** ([lib/api.ts](lib/api.ts)):
- `request()` now catches fetch failures, distinguishes `AbortError` from generic network failures, and throws a clear `ApiError('TIMEOUT'|'NETWORK_ERROR', …)` instead of letting the raw abort propagate.
- Added a `scrape?: boolean` option mirroring mobile's tiering: 45s (`SCRAPE_TIMEOUT_MS`) for all `/api/integrations/grades/*` calls (`portalLoginHAC`, `portalLoginPS`, `portalSyncProfile`, `portalGrades`, `portalTranscript`, `portalSchedule`, `portalClasswork`, `portalReportCard`, `portalGpa`, `portalProgressReport`, `portalAttendance`, `portalContactTeachers`), 30s (`DEFAULT_TIMEOUT_MS`, unchanged) for everything else.
- **Did not touch** the HAC login POST flow or origin-extraction logic itself (`backend/src/integrations/grades/hacClient.ts`) — only the *client-side* HTTP timeout/error-handling wrapper, which is a different layer.

## College Feature Regression

**Verified, immediately-reproducible cause: the Python `model_server/` (FastAPI microservice) that `/colleges/predict` and `/colleges/path` depend on is not running.** `npm run dev` at the repo root only starts the Next.js web app + Express backend (per ARCHITECTURE.md) — it never starts `model_server/`, and there's no entry for it in `.claude/launch.json` either. Confirmed directly in this session: `curl http://localhost:8001/health` returns connection-refused (nothing listening on port 8001, the `MODEL_SERVER_URL` default). Every `/colleges/predict` and `/colleges/path` call therefore hits the `axios.post(...)` in [backend/src/services/collegeProbability.ts:64-93](backend/src/services/collegeProbability.ts:64), gets `ECONNREFUSED`, and throws `ModelServiceUnavailableError` — which the route correctly catches and turns into a clean `503 "Prediction service is temporarily unavailable"`. From a user's perspective this looks exactly like "the College feature is broken": every college shows "No prediction available," the "path to admission" section always errors. **To fix: start the model server separately** (e.g. `cd model_server && pip install -r requirements.txt && uvicorn main:app --port 8001`, or whatever its actual entrypoint is) alongside `npm run dev`. Worth adding to `.claude/launch.json` and to ARCHITECTURE.md, which doesn't mention this new Python component at all yet.

**A second, related but independently-real gap found while investigating:** the `College` / `CollegeListItem` / `CollegePathCache` Prisma models were added to `schema.prisma` in commit `9ccc5f2`, but no database migration was ever committed for them (see evidence below). Directly querying this session's live DB shows the tables **do already exist** (with one pre-existing `CollegeListItem` row) — so on this specific database, someone already ran a schema-sync (most likely `prisma db push`) out-of-band at some point, and this gap is currently latent rather than actively broken here. It's still a real defect worth closing, because (a) it means Prisma's own migration history doesn't match the live schema — a future `prisma migrate deploy` could report drift — and (b) on any other environment (a fresh clone, a new Neon branch, a from-scratch Vercel deploy) these tables genuinely would not exist yet, and would reproduce exactly the hang described below.

Evidence:
1. `git show --stat 9ccc5f2 -- backend/prisma/migrations/` shows only `20260705000000_add_coppa_fields/migration.sql` (unrelated COPPA columns) touched by that commit — **no migration for the College tables was committed.**
2. A migration folder exists on disk at `backend/prisma/migrations/20260706000000_add_college_catalog/`, but `migration.sql` inside it is an **empty directory**, not a SQL file (0 bytes, no content) — `git ls-tree -r HEAD -- backend/prisma/migrations/20260706000000_add_college_catalog` returns nothing, confirming this was **never committed to git** (git can't track empty directories). It's a stray, broken leftover, likely from an interrupted/incomplete migration attempt.
3. `backend/package.json`'s `dev` script is `ts-node-dev --respawn src/index.ts` — it does **not** run `prisma migrate deploy`. Only `npm start` (`prisma migrate deploy && node dist/src/index.js`) does. So the normal local dev loop (`npm run dev` at repo root) never applies pending migrations at all.
4. `ensureSchema()` ([backend/src/lib/startup.ts](backend/src/lib/startup.ts)) — the self-healing raw-SQL patcher that the *previous* commit (`0e8f95e`) identified as the mechanism production actually relies on, since Vercel's serverless cold starts can't reliably run `migrate deploy` — was **only wired into `backend/api/index.ts`** (the Vercel entrypoint), never into `backend/src/index.ts` (local dev), **and its patch list had no entry for `College`/`CollegeListItem`/`CollegePathCache` at all.**

On any environment where the migration truly hasn't been synced (a fresh clone, a new Neon branch, a from-scratch Vercel deploy), any request touching `prisma.college`, `prisma.collegeListItem`, or `prisma.collegePathCache` would throw a Postgres "relation does not exist" error.

**A real compounding bug regardless of table existence, found while tracing this:** `GET /colleges` and `DELETE /colleges/:id` in [backend/src/routes/colleges.ts](backend/src/routes/colleges.ts) (pre-fix, lines 232-239 and 258-267) had **no `try/catch`** around their Prisma calls. Express's global error handler ([backend/src/app.ts:402-419](backend/src/app.ts:402)) explicitly only catches errors passed to `next(err)` or thrown inside *non-async* handlers (per its own comment) — it does **not** catch a rejected promise from an async handler that never calls `next()`. So on any database error (missing table, connection blip, etc.) these two routes wouldn't return a 500; they'd simply hang with no response until the client's own timeout aborted the request — which is exactly the mechanism that would produce a "signal aborted" symptom for this feature specifically, on whatever environment does hit a DB error here. (`POST /colleges/predict` and `/colleges/path` already had a catch-all `try/catch`, so those degrade to a clean 500/503 rather than hanging — confirmed above with the model-server-down case.)

**Fixes applied:**
- [backend/src/routes/colleges.ts](backend/src/routes/colleges.ts): wrapped `GET /` and `DELETE /:id` in `try/catch`, matching the pattern already used in `/predict` and `/path` — a DB error now returns a clean 500 instead of hanging.
- [backend/src/lib/startup.ts](backend/src/lib/startup.ts): added `CREATE TABLE IF NOT EXISTS` patches for `College`, `CollegeListItem`, and `CollegePathCache` (matching `schema.prisma` field-for-field) to `ensureSchema()`'s `PATCHES` list, and extended the existence-probe to check for all three new tables (the probe short-circuits and skips every patch if it doesn't check the newest addition — the file's own comment flags this as something to keep in sync).
- [backend/src/index.ts](backend/src/index.ts): now calls `ensureSchema()` on startup (fire-and-forget, matching the Vercel entrypoint's pattern) so local dev self-heals the same way production does, instead of only production ever getting the patch.

**Not fixed — needs your decision:** the real, durable fix is a proper hand-authored `migration.sql` in `backend/prisma/migrations/20260706000000_add_college_catalog/` (replacing the stray empty directory) so `prisma migrate deploy` and Prisma's migration history stay authoritative. `backend/prisma/migrations/**` is on this project's hard-protected path list (`.claude/guardrails/protected-paths.json`) — Edit/Write there is denied outright regardless of task scope, and deliberately bypassing that via Bash isn't something I'll do without you saying so, since it's a live Neon Postgres. The `ensureSchema()` patch above unblocks the feature functionally in the meantime. 2. **`.env` git hygiene** — `.gitignore` has `.env*` with `!.env.example` — real env files are excluded, example file is intentionally committed. Good.

4. **CORS** — [backend/src/app.ts:86-119](backend/src/app.ts:86) uses an explicit `ACTIVE_ORIGINS` allowlist (env-configured `ALLOWED_ORIGINS`, falling back to a fixed dev list) with `credentials: true`. Not a wildcard-with-credentials misconfiguration.
5. **Rate limiting** — `/auth/register` and `/auth` have dedicated limiters; `/integrations/grades/hac/login`, `/powerschool/login`, and `/sync-profile` have `portalLimiter`; `/colleges/predict` and `/colleges/path` have their own limiters (`collegePredictLimiter`/`collegePathLimiter`).
